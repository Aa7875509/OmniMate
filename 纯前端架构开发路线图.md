### OmniMate 纯前端架构开发路线图

#### 阶段一：骨架搭建与核心通信（预计耗时：1-2周）

此阶段的目标是“让应用跑起来”，重点在于构建 Electron 的壳，打通主进程与渲染进程的通信管道，并确立 Vue 3 的 UI 框架。这是整个项目的地基，决定了后续开发的顺畅程度。

你需要首先初始化项目脚手架。使用 Vite 创建 Vue 3 项目，并集成 Electron Builder 或 Electron Vite 来管理构建流程。这一步的关键是配置好 TypeScript，因为 OmniMate 涉及复杂的逻辑交互，强类型能极大减少后期维护成本。你需要划分清晰的目录结构，将 `main`（主进程）、`preload`（预加载脚本）和 `renderer`（渲染进程）物理隔离。

接下来是核心的 IPC 通信层开发。在 Electron 中，为了安全起见，不能直接在渲染进程调用 Node.js API。你需要利用 `contextBridge` 暴露安全的 API 接口。定义一套标准的 `ipcHandle` 协议，例如 `invoke-chat`、`on-audio-data` 等。编写一个简单的测试用例：在 Vue 界面点击按钮，主进程控制台打印日志，验证双向通信是否通畅。

最后是基础 UI 框架的搭建。引入 TailwindCSS 实现你设计的“以太玻璃”风格，封装基础的 GlassPanel 组件。此时不需要真实的 3D 模型，用一个旋转的立方体占位即可，重点是实现窗口的拖拽区域定制和无框窗口的美观度。

**里程碑交付物**：一个可以打包运行的 .exe/.dmg 安装包，打开后是一个无框的毛玻璃窗口，点击按钮能与后台进行数据交互。

#### 阶段二：大脑接入与对话流（预计耗时：2-3周）

这一阶段要让 OmniMate“活”过来，能够进行文本层面的智能对话。我们将暂时忽略语音和视觉，专注于 LLM 的接入和上下文管理。

首先在 Node.js 主进程中实现 AI 适配器模式。创建一个统一的 `LLMProvider` 接口，然后分别实现 `OpenAIProvider` 和 `LocalLlamaProvider`（基于 `node-llama-cpp`）。这意味着无论用户选择云端 API 还是本地 GGUF 模型，上层业务代码只需要调用统一的 `chat()` 方法。你需要在这个阶段完成“模型管理中心”的前端界面，允许用户输入 API Key 或选择本地模型文件路径，并将这些配置持久化存储在 `electron-store` 中。

其次是上下文管理器的实现。在 Node.js 内存中维护一个数组作为滑动窗口，存储最近的对话历史。每次发送请求前，将该数组拼接到 Prompt 中。注意处理 Token 溢出问题，当历史记录过长时，自动剔除最早的对话。

最后是渲染进程的聊天界面开发。实现类似微信或 ChatGPT 的气泡布局，支持 Markdown 渲染（使用 `markdown-it`），并加入代码高亮功能。确保输入框支持多行输入，且回车键能正确触发发送。

**里程碑交付物**：用户可以在设置中选择模型，在主界面与 AI 进行流畅的文字问答，AI 能记住前几轮的对话内容。

#### 阶段三：听觉与口语觉醒（预计耗时：3-4周）

这是从“聊天软件”向“智能助手”跨越的关键一步。我们需要引入音频处理能力，实现全双工的语音交互。

首先是麦克风采集与 VAD。在 Electron 中使用 `microphone-stream` 或原生录音模块获取 PCM 音频流。为了实现“随时打断”和“自动停止”，你需要集成 VAD。考虑到纯 JS 实现的复杂度，建议初期使用简单的音量阈值检测，或者在 Node.js 中引入 `onnxruntime-node` 运行 Silero VAD 模型。当检测到有效人声时开始录音，静音超过阈值则自动停止并发送。

其次是 ASR 与 TTS 的链路打通。录音结束后，将音频 Buffer 发送给 Whisper（API 或本地）。获得文本后，立即触发 LLM 回复。LLM 生成文本的同时，流式地将文本发送给 TTS 服务（如 Edge-TTS 或 CosyVoice）。这里有一个技术难点：Node.js 生成的音频流需要传输到前端播放。你可以利用 `child_process` 启动一个 FFmpeg 子进程将音频流转码为浏览器兼容格式，或者直接通过 IPC 传递 Buffer 并在前端使用 AudioContext 解码播放。

最后是简单的口型同步。在前端监听音频播放的时间点，分析音频波形振幅。将这个振幅数值映射到 3D 模型的嘴巴张开权重上。虽然不如音素级同步精准，但足以产生“说话”的视觉效果。

**里程碑交付物**：用户可以按住说话或直接对话，OmniMate 能听写文字、思考并朗读出答案，同时 3D 模型的嘴巴会随声音张合。

#### 阶段四：视觉感知与多模态融合（预计耗时：2-3周）

赋予助手“眼睛”。这一步主要涉及硬件调用和图片数据处理。

首先是摄像头快照功能。使用 `fluent-ffmpeg` 调用系统摄像头。你需要编写一个预览组件，让用户确认摄像头画面。当触发“看图”指令时，截取当前帧。

其次是图片预处理。大模型对输入图片的大小有限制。使用 `sharp` 库在 Node.js 中将截取的帧压缩为 JPEG，调整分辨率至 640x640 左右，并将其转换为 Base64 或二进制流。

最后是多模态 Prompt 构建。修改之前的 `LLMProvider`，使其支持 `image` 参数。当用户上传图片或触发拍照时，构建包含 `content: [{ type: 'text', text: '...' }, { type: 'image_url', image_url: {...} }]` 格式的请求体。在 UI 上，当助手正在“看”图时，数字人应表现出注视前方的专注神态。

**里程碑交付物**：用户拿起一个苹果问“这是什么”，OmniMate 能通过摄像头看到并回答“这是一个苹果”。

#### 阶段五：数字人灵魂注入与性能调优（预计耗时：2周+）

最后的打磨阶段，专注于体验优化和性能稳定性。

首先是 3D 表现力的升级。引入 Three.js 的动画混合器，根据 LLM 返回的情感标签（如 [开心]、[疑惑]），触发生硬的 Idle 动画切换到对应的肢体动作。优化口型算法，如果可能，尝试接入更精准的 Audio2Face 方案。

其次是性能监控与优化。Electron 吃内存是出了名的。你需要利用 Node.js 的 `process.memoryUsage` 监控主进程内存，防止泄漏。对于本地模型推理，确保它在一个独立的线程或子进程中运行，避免阻塞 GUI 渲染线程，导致界面卡死。

最后是打包与分发。配置 `electron-builder`，制作静默安装程序。针对本地模型用户，考虑如何将模型文件与程序分离，支持首次运行时下载模型或指定已有模型路径。

**里程碑交付物**：一个体验流畅、反应灵敏、具备情感反馈的桌面版 OmniMate 正式版。
