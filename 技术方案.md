### OmniMate 纯前端架构技术方案

#### 核心架构理念：Electron 即后端

我们将 Electron 的主进程作为“超级后端”，它直接管理硬件资源、调用 AI 接口、处理音视频流；渲染进程则专注于高性能的 3D 渲染和用户交互。

**技术栈概览**

| 层级 | 技术选型 | 职责描述 |
| ------ |------ |------ |
| **用户界面层** | Vue 3 + Vite + TailwindCSS | 构建设置面板、聊天窗口、仪表盘 |
| **3D 表现层** | Three.js + @vueuse/core | 负责数字人模型加载、骨骼动画、口型同步渲染 |
| **IPC 通信层** | Electron IPC (ContextBridge) | 安全地打通渲染进程与主进程的指令传输 |
| **业务逻辑层** | Node.js (Main Process) | 核心中枢，负责任务编排、状态机管理 |
| **AI 适配层** | OpenAI SDK, node-llama-cpp | 统一封装云端 API 和本地 GGUF 模型推理 |
| **多媒体层** | fluent-ffmpeg, sharp, mic | 摄像头采集、音频转码、图片压缩预处理 |

#### 模块一：AI 模型管理中心

这是系统的“大脑皮层”，允许用户在设置中动态配置不同的模型来处理听、看、说、想。

**1. 本地模型集成**
使用 `node-llama-cpp` 库，直接在 Electron 主进程中加载量化后的 GGUF 模型。

- **优势**：无需配置复杂的 Python 环境，用户只需下载模型文件即可运行。
- **实现**：在 `main.js` 中初始化 `LlamaModel` 实例，通过 `LlamaContext` 进行对话补全。

**2. 远程 API 路由**
使用 `openai` 官方 SDK（兼容 DeepSeek、Claude 等标准）。

- **实现**：创建一个通用的 `ChatClientFactory`，根据用户配置的 `BaseURL` 和 `API Key` 动态生成客户端实例。

**3. 视觉模型处理**
对于多模态能力（看图），优先使用支持 Vision 的 API 模型（如 GPT-4o, Qwen-VL）。如果是本地模型，需确保选用的 GGUF 版本支持图像输入（如 LLaVA 格式的 GGUF）。

#### 模块二：感知系统

由于没有独立的后端，我们需要在 Electron 主进程中直接操作硬件流。

**1. 听觉通道**

- **采集**：使用 `electron-microphone` 或原生的 `AudioRecorder` 获取麦克风 PCM 数据流。
- **语音识别**：
    - **云端方案**：将 PCM 流通过 WebSocket 发送给 Whisper API。
    - **本地方案**：使用 `whisper-node` 绑定本地 C++ 库，在主进程中直接运行 ASR 推理。

**2. 视觉通道**

- **采集**：使用 `fluent-ffmpeg` 调用系统摄像头，截取特定帧。
- **预处理**：使用 `sharp` 库在内存中将视频帧压缩为 JPEG/Base64，控制分辨率（例如 512x512），以减少传给大模型的 Token 消耗。

#### 模块三：认知与决策引擎

这是一个运行在 Node.js 主进程中的状态机。

**1. 上下文管理器**
维护一个基于数组的滑动窗口记忆。每次对话前，将最近 N 轮对话历史拼接到 System Prompt 中。

**2. 意图识别与工具调用**
当用户提问时，LLM 首先判断意图：

- **闲聊**：直接生成回复。
- **看图**：触发摄像头截图 -> 拼接图片与文本 -> 调用视觉模型。
- **查询**：调用本地搜索或知识库。

#### 模块四：表达与渲染

这是渲染进程的核心工作。

**1. 语音合成**

- **流式播放**：Node.js 主进程从 TTS 接口获取音频流，通过 IPC 分块传输给渲染进程，使用 HTML5 Audio 标签进行无缝播放。

**2. 3D 数字人驱动**

- **口型同步**：
    - **简易版**：在渲染进程中分析当前播放音频的频率数据，映射到 3D 模型的嘴巴张开度。
    - **进阶版**：如果 TTS 接口返回音素时间戳，渲染进程根据时间轴精确控制 BlendShapes。
- **肢体语言**：解析 LLM 返回的情感标签（如 `[开心]`），触发 Three.js 的动画混合器播放对应动作。

### 开发路线图

建议分为三个阶段，逐步迭代，确保每个阶段都有可运行的交付物。

#### 第一阶段：原型期——“会说话的盒子”

目标：跑通 Electron 主进程与渲染进程的通信，实现基本的文字问答。

**关键任务**

- **项目脚手架搭建**：配置 Vite + Electron + Vue 3 开发环境。
- **IPC 通道建立**：定义 `ipcHandle` 和 `ipcRenderer` 的安全通信协议。
- **模型接入**：在 Node.js 中实现 OpenAI 协议的标准调用，并在 UI 上做一个简单的输入框。
- **基础 3D 场景**：在 Vue 中引入 Three.js，加载一个简单的静态模型（如立方体或导入的 GLTF 模型）。

**验收标准**
用户在输入框打字，Node.js 调用 API 返回结果，界面显示文字，且 3D 模型能简单变色表示“正在思考”。

#### 第二阶段：成长期——“视听感知觉醒”

目标：赋予助手听和看的能力，并加入语音交互。

**关键任务**

- **麦克风接入**：实现录音功能，并将音频数据发送给 Whisper（API 或本地）。
- **TTS 集成**：接入 CosyVoice 或其他 TTS 服务，实现文本转语音并在前端播放。
- **摄像头快照**：实现“拍照”按钮，调用 `fluent-ffmpeg` 截取画面，并使用 `sharp` 压缩。
- **多模态 Prompt 构建**：编写 Prompt 模板，将图片和文字组合发送给支持 Vision 的模型。

**验收标准**
用户可以按住说话，助手能转写文字并语音回复；用户可以打开摄像头拍一个物体，问“这是什么”，助手能准确回答。

#### 第三阶段：成熟期——“智能管家”

目标：完善 3D 表现力，增加本地模型支持，优化性能。

**关键任务**

- **本地推理集成**：引入 `node-llama-cpp`，允许用户选择本地 GGUF 文件进行离线对话。
- **3D 表情驱动**：实现音频波形到 BlendShapes 的映射算法，让数字人嘴巴随声音开合。
- **模型管理中心 UI**：开发完善的设置页面，允许用户添加 API Key、切换模型、调整温度参数。
- **性能优化**：优化 Node.js 的事件循环，防止繁重的 AI 计算阻塞 UI 渲染。

**验收标准**
拥有一个完整的设置面板；支持一键切换云端/本地模型；数字人在说话时有自然的口型和眨眼动作；整体运行流畅，无明显卡顿。
